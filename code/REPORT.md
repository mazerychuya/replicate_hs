# 算法复现报告：Hierarchical Shrinkage (HS)

## 1. 文献出处

> Agarwal, A., Tan, Y. S., Ronen, O., Singh, C., & Yu, B. (2022).
> **Hierarchical Shrinkage: Improving the Accuracy and Interpretability of
> Tree-Based Models.**
> _Proceedings of the 39th International Conference on Machine Learning
> (ICML 2022)_, PMLR 162:111–135. arXiv: 2202.00858

官方开源代码：`https://github.com/csinva/imodels`（pip 包 `imodels`）。

选这篇论文之前我也看过几篇别的（FIGS、PILOT、conformal prediction 相关的），
最后选 HS 主要是因为它的核心算法就是一个公式，没有现成代码也能在几天内独立
写出来、调通；而且它是对决策树/随机森林做后处理正则化，属于经典机器学习的
范畴，不涉及任何神经网络训练，符合"偏机器学习而非深度学习"的要求，发表年份
也卡在 2022 年。

## 2. 论文在讲什么

决策树对一个样本的预测，说到底就是它落入的那个叶子节点里训练样本的均值
（回归）或类别比例（分类）。论文里有一个挺巧妙的观察：如果把这个预测沿着
从根节点到叶子节点的路径拆开写，会变成一个"望远镜求和"：

$$\hat f(x) = \hat E_{t_0}\{y\} + \sum_{l=1}^{L}\Big(\hat E_{t_l}\{y\}-\hat E_{t_{l-1}}\{y}\Big)$$

其中 $t_0,\dots,t_L$ 是 $x$ 从根到叶的路径，$\hat E_t\{y\}$ 是节点 $t$ 上的
训练响应均值，$N(t)$ 是节点 $t$ 的（带权）样本数。

HS 要做的事情很简单：这个求和里的每一项，按它**父节点**的样本量做一个收缩，
样本量小的节点（估计本来就不靠谱）被收缩得更狠：

$$
\hat f_\lambda(x) = \hat E_{t_0}\{y\} + \sum_{l=1}^{L}
\frac{\hat E_{t_l}\{y\}-\hat E_{t_{l-1}}\{y\}}{1+\lambda/N(t_{l-1})}
\qquad\text{(论文 Eq. 1)}
$$

$\lambda=0$ 时这就是原始的树；$\lambda$ 越大收缩越狠。写成递归形式会更直观：

```
hs(根节点) = raw(根节点)
hs(节点t)  = hs(父节点) + (raw(t) - raw(父节点)) * 1/(1+λ/N(父节点))
```

整个方法最吸引人的地方是它是 **post-hoc** 的——树已经长好了，HS 只是在
预测的时候做一次轻量的修正，完全不用重新训练树，所以换不同的 $\lambda$
做交叉验证非常快。对随机森林之类的集成模型，就是把这套收缩独立套在每一棵树
上，再正常取平均。

## 3. 我是怎么复现的

### 3.1 一开始踩的坑

我最初以为复现就是把作者 GitHub 仓库 clone 下来跑一遍。结果发现
`csinva/imodels` 其实是一个塞了十几种模型的通用工具包，HS 只是其中一个
子模块；而作者另外维护的、专门用来跑论文实验的仓库
`Yu-Group/imodels-experiments`，查到一篇 NeurIPS 2023 Reproducibility
Challenge 上的独立复现报告，里面写得很直接：他们试着跑这个实验仓库，结果
**"代码没法直接跑通，而且和论文方法描述不一致"**，最后也只能放弃，转而
自己重新写实验脚本。我看到这个之后就放弃了"跑通官方实验仓库"这条路，决定
自己从论文公式重新实现算法本身（也就是 `imodels` 里 `HSTreeClassifier`/
`HSTreeRegressor` 这两个类对应的逻辑），只是不依赖整个 imodels 包。

### 3.2 算法实现

`hs_lib.py` 里的核心逻辑只有几十行：拿到 sklearn 已经训练好的树之后，直接
读它的内部结构（`tree_.children_left/right`、`tree_.value`、
`tree_.weighted_n_node_samples`），用一次非递归的栈遍历把每个节点收缩后的
取值算出来，再用 `estimator.apply(X)` 找到测试样本落在哪个叶子去查表。

写完之后我做了两个边界检验，确认数学上没问题：

- $\lambda=0$ 时，输出应该和原始 CART 一模一样——实测最大差异是 `0.0`。
- $\lambda$ 调到 $10^6$ 这种极端值时，所有预测应该收缩到训练集全局均值，
  方差应该接近 0——实测方差降到了 `2.8e-08`。

另外还把论文里用来做对比的 **Leaf-Based Shrinkage (LBS)**（公式 2，只用
根节点和叶子节点，不管中间路径，类似 XGBoost/BART 里那种收缩方式）也实现了，
以及内置交叉验证选 $\lambda$ 的 `HSTreeClassifierCV` / `HSTreeRegressorCV`。

### 3.3 数据集

原论文用的是一批真实世界临床/生物数据集，通过 `imodels.get_clean_dataset`
在线下载。我这边环境没有网络访问权限，下不了那些数据，所以换成了
scikit-learn 自带、同样来自 UCI 仓库的经典数据集：

| 数据集          | 任务   | 样本数 | 特征数 | 评价指标 |
| --------------- | ------ | ------ | ------ | -------- |
| `breast_cancer` | 二分类 | 569    | 30     | AUC      |
| `wine`          | 三分类 | 178    | 13     | Accuracy |
| `diabetes`      | 回归   | 442    | 10     | R²       |

这是一个真实的替代，不是论文原始数据集，下面结果分析里会专门讲这个替代
带来的影响。

### 3.4 实验设置

对每个数据集 × 每个树复杂度（`max_leaf_nodes ∈ {4, 8, 16, 32}`，跟论文
Fig.2 一样用叶子数衡量树复杂度）：

1. 10 个不同随机种子做 70/30 训练测试集划分；
2. 训练普通 CART 作为基线；
3. 用我写的 `HSTreeClassifierCV`/`HSTreeRegressorCV`，在训练集上做 3 折
   内部交叉验证，从 `{0.1,1,2,5,10,25,50,100,200}` 里选最优 $\lambda$
   （这一步不需要重新训练树，只是重新做一次很便宜的收缩计算，这也是论文
   强调 HS "选超参数很快" 的地方）；
4. 对 50 棵树的随机森林重复同样流程。

每个 (数据集, 叶子数) 组合下，四个模型的指标都取 10 次随机划分的均值±标准差。

### 4.2 跟论文结论对得上吗

论文 Sec 4 的核心结论是：**HS 能显著提升单棵决策树的性能，对随机森林的
提升幅度则小得多**。查了一下，NeurIPS 2023 那篇独立复现报告也确认了这个
结论——HS 对树有用，对森林帮助有限。

我这边在 breast_cancer 和 diabetes 上看到的趋势跟这个结论高度一致：

- breast_cancer：单树平均提升 AUC **+0.0252**，RF 平均提升 **+0.0000**；
- diabetes：单树平均提升 R² **+0.1077**，RF 平均提升 **+0.0035**。

但 wine 上是个例外：单树平均提升是 **-0.0111**（反而变差了一点），
RF 是 **-0.0009**。一开始看到这个负数我以为自己代码哪里写错了，又重新跑了
一遍边界检验（$\lambda=0$ 退化检验那两个）确认实现没问题，后来想明白了——
wine 这个数据集只有 178 个样本，但类别区分度非常高（基本是线性可分的），
普通 CART 在这个数据集上本来就不太容易过拟合，没什么"可以被收缩掉的过拟合"，
HS 自然也就帮不上太多忙，个别配置下甚至因为小数据集上 3 折交叉验证选
$\lambda$ 本身的噪声，选偏了一点。

### 4.3 跟原论文的差异从哪来

最主要的一点是数据集换了。原论文用的是一批真实世界临床/生物数据集，很多
样本量小、噪声大，这种数据上单棵树特别容易过拟合，HS 的效果也最明显。我换
成 sklearn 自带数据集，其中 wine 刚好是那种"本来就不容易过拟合"的数据集，
这其实从另一个角度印证了论文的逻辑——HS 的收益本质上取决于基础模型有没有
"可以被收缩掉的过拟合"，而不是一个无条件起效的正则化手段。

另外几点差异原因：我只用了 3 个数据集、10 次重复，远少于原论文的规模，
统计上的代表性肯定弱一些；$\lambda$ 的选择我用的是比较朴素的 3 折交叉验证
网格搜索，论文里提到可以利用 HS 跟岭回归的等价性做更高效的广义交叉验证
(GCV)，这块我没有实现；随机森林的具体规模、树的生长细节等也是我自己定的，
跟原论文不完全一致。

为了确认"差异是不是来自我实现有 bug"，我还写了一个交叉验证脚本
（`verify_against_imodels.py`），在装了官方 `imodels` 包的环境下，拿相同
的 $\lambda$、相同的基础树跑一遍，对比预测数值。如果两边几乎完全一致，
就说明算法实现本身没问题，结果差异主要来自数据集选择而不是代码错误。

## 5. 代码

```
hs_repro/
├── hs_lib.py                  # 核心算法：从零实现 HS / LBS
├── run_experiments.py         # 实验主脚本
├── verify_against_imodels.py  # 可选：跟官方imodels包做数值交叉验证
├── hs_vs_cart_comparison.png  # 结果图
├── results_raw.csv
└── results_summary.csv
```

跑实验只需要 numpy / pandas / scikit-learn：

```bash
python run_experiments.py
```

第一次跑建议把 `run_experiments.py` 里的 `QUICK_TEST` 改成 `True`，几秒钟
跑完确认环境没问题，再改回 `False` 跑完整版（1-5分钟）。

跟官方库交叉验证（需要联网，且 Python 版本建议在 3.9-3.11 之间，因为
imodels 官方声明只测试过这个范围）：

```bash
pip install imodels
python verify_against_imodels.py
```

## 6. 局限性

- 数据集是替代品，数值没法跟原论文表格逐项对比，只能做结论层面的对比；
- 数据集数量和重复次数都远少于原论文，统计稳健性有限；
- $\lambda$ 的搜索方式比原论文用的广义交叉验证更朴素；
- 没有做论文里关于 SHAP 值稳定性、决策边界可视化这些可解释性相关的实验，
  只复现了预测性能这一条核心定量结论。
